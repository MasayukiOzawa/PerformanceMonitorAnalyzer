# 64-グラフ領域の表示調整

## タスクの概要
グラフデータ領域とデータテーブル領域の境界でグラフ表示領域を+100 高さを増やす修正を実施しました。

## 実施した作業

### 1. リポジトリ構造の理解
- C# WPFアプリケーション（Performance Monitor Analyzer）
- ScottPlot.WPFライブラリを使用したグラフ機能
- Windows専用アプリケーション（Linux環境でのビルド不可）

### 2. UI レイアウト構造の分析
MainWindow.xamlのGrid.RowDefinitions構造：
- Row 0: メニューバー（Auto）
- Row 1: プログレスバー（Auto）  
- Row 2: カウンター選択とグラフ表示の上部エリア（*）
- Row 3: GridSplitter（5ピクセル）
- Row 4: データテーブルとログ表示の下部エリア（元：550ピクセル）

### 3. 修正内容
**ファイル**: `/src/PerformanceMonitorAnalyzer/MainWindow.xaml`
**変更行**: 15行目

```diff
- <RowDefinition Height="550" MinHeight="100" MaxHeight="750"/>
+ <RowDefinition Height="450" MinHeight="100" MaxHeight="750"/>
```

### 4. 変更の影響
- グラフ表示領域（Row 2）：Height="*"により残りのスペースを使用 → +100ピクセル増加
- データテーブル領域（Row 4）：Height="550" → "450" → -100ピクセル減少
- MinHeight（100）とMaxHeight（750）は保持：ユーザーによる手動調整は引き続き可能

## 技術的な詳細

### WPFのGridレイアウトシステム
- `Height="*"`：利用可能な残りスペースを使用
- `Height="450"`：固定サイズ（450ピクセル）
- GridSplitterにより、実行時にユーザーが境界をドラッグして調整可能

### 設計上の考慮事項
- MinHeight="100"により、データテーブル領域が完全に隠れることを防止
- MaxHeight="750"により、データテーブル領域が過度に大きくならないよう制限
- GridSplitterの動作は変更せず、ユーザビリティを維持

## 検証方法
Linux環境でのWPFビルドは不可のため、以下で検証：
1. XAMLの構文チェック（正常）
2. Git diffによる変更内容確認（適切）
3. 論理的な影響分析（期待通り）

Windows環境での実際の動作確認が推奨されます。

## 結果
- グラフ表示領域が100ピクセル高くなり、より詳細なデータ可視化が可能
- データテーブル領域は450ピクセルで十分な表示領域を維持
- ユーザーによる手動調整機能は保持
